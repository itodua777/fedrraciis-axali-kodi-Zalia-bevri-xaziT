# SaaS Multi-Tenant Backend (NestJS + Prisma + CASL) Implementation Plan

ეს დოკუმენტი აღწერს Fitness SaaS პლატფორმის ბექენდის არქიტექტურულ გაუმჯობესებას, შეცდომების გასწორებას და მასშტაბირებადი, სენიორ დონის სისტემის ჩამოყალიბების გეგმას.

---

## 1. მიმდინარე მდგომარეობა და აღმოჩენილი ხარვეზები (Gaps)

1. **პრიზმის სქემის ვალიდაციის პრობლემა (გასწორებულია)**:
   - `Department` მოდელი მთლიანად აკლდა `prisma/schema` ფოლდერს, თუმცა მასზე მიმართვები არსებობდა სხვა ფაილებში. შევქმენით `department.prisma` და სქემის ვალიდაცია წარმატებით გადის.
   
2. **CLS (Continuation Local Storage) კონტექსტის ორდერინგი**:
   - `ClsModule` middleware მუშაობს NestJS-ის Guard-ებამდე. შესაბამისად, როდესაც middleware ეშვება, `req['user']` ჯერ კიდევ `undefined`-ია, რადგან JWT Guard-ს ავტორიზაცია ჯერ არ გაუვლია.
   - **შედეგი**: Prisma Middleware ვერ ხედავს იუზერს და `companyId`-ს ავტომატურ ფილტრაციას ვერ აკეთებს.

3. **Tenant Leakage (მულტი-ტენანტობის უსაფრთხოების რისკი)**:
   - მიმდინარე Prisma Middleware ფილტრავს მხოლოდ `findUnique`, `findFirst`, `findMany` და `count` ოპერაციებს.
   - **პრობლემა**: `create`, `update`, `delete` ოპერაციები არ კონტროლდება. იუზერს შეუძლია სხვა კომპანიის `companyId`-ით ჩაწეროს, შეცვალოს ან წაშალოს მონაცემები.

4. **CASL dynamic permissions და Performance (მასშტაბირება)**:
   - `PoliciesGuard` ყოველ რექვესტზე ითხოვს იუზერის როლების და უფლებების შემოწმებას. JWT payload-ში მხოლოდ ID-ები გვაქვს.
   - **პრობლემა**: თუ ყოველ HTTP რექვესტზე ბაზიდან ამოვიღებთ იუზერის როლებს და უფლებებს, მაღალ დატვირთვაზე (at scale) მონაცემთა ბაზა გადაიტვირთება. საჭიროა ქეშირების ფენის (Cache Layer) დანერგვა.

---

## 2. არქიტექტურული გეგმა ნაბიჯ-ნაბიჯ

ჩვენს გეგმას დავყოფთ 5 მთავარ ნაბიჯად.

### ნაბიჯი 1: CLS და Auth Guard-ის სინქრონიზაცია
ჩავასწორებთ `UserAccessTokenGuard`-ს და `clsOptions`-ს, რომ ავტორიზაციის წარმატებით გავლის შემდეგ, მომხმარებლის ობიექტი ჩაიწეროს CLS-ში. ეს უზრუნველყოფს, რომ Prisma-ს სერვისმა ნებისმიერ მომენტში იცოდეს მიმდინარე მომხმარებლის და კომპანიის მონაცემები.

### ნაბიჯი 2: სრული Multi-Tenant უსაფრთხოების უზრუნველყოფა (Prisma Level)
მოვახდენთ Prisma-ს რეპოზიტორების ან მიდლვეარის მოდიფიკაციას, რათა:
1. `create` ოპერაციისას ავტომატურად მიენიჭოს CLS-ში არსებული `companyId`.
2. `update` და `delete` ოპერაციებისას მოხდეს ვალიდაცია, რომ რესურსი ნამდვილად ეკუთვნის მიმდინარე კომპანიას.
3. თავიდან ავიცილოთ ტენანტებს შორის მონაცემების გაჟონვა (Tenant Isolation).

### ნაბიჯი 3: უფლებების ქეშირების სისტემა (Permission Caching Layer)
მონაცემთა ბაზის დასაცავად და სისწრაფის მისაღწევად:
1. შევქმნით `PermissionsCacheService`-ს (ამ ეტაპზე memory cache, რომელიც მომავალში მარტივად ჩანაცვლდება Redis-ით).
2. ავტორიზაციისას/უფლებების შემოწმებისას მონაცემებს წავიკითხავთ ქეშიდან.
3. როდესაც კომპანიის ადმინისტრატორი შეცვლის როლს ან უფლებებს, მოვახდენთ ქეშის ინვალიდაციას (Cache Invalidation).

### ნაბიჯი 4: CASL Ability Factory და SuperUser ლოგიკის დახვეწა
1. `isSuperUser` მომხმარებლებისთვის სრული უფლებების მინიჭება (`manage` `all`) პირდაპირ კოდის დონეზე.
2. CASL JSON პირობების (conditions) და ველების ფილტრაციის (fields) სრული მხარდაჭერა.
3. `PoliciesGuard`-ისა და `@CheckPolicies` დეკორატორის გამართვა, რათა კონტროლერებში მარტივად და უსაფრთხოდ გამოვიყენოთ.

### ნაბიჯი 5: სრული ტესტირება (E2E & Unit)
დავწერთ სატესტო სკრიპტებს, რომლებიც შეამოწმებენ:
1. კომპანიის რეგისტრაციას და როლების დინამიურ შექმნას.
2. სხვადასხვა უფლებების მქონე იუზერების ავტორიზაციას.
3. ტენანტების იზოლაციას (მცდელობა სხვა კომპანიის მონაცემის წაკითხვის/შეცვლის).
4. უფლებების დინამიურ შეცვლას და ქეშის მომენტალურ განახლებას.

---

## 3. მუშაობის დაწყება

> [!IMPORTANT]
